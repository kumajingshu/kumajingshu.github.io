Metric Tracking Platform

Projects' metric performance tracking and dashboarding web app for a top tech company

Interaction Design

Introduction

In the first half of 2020, I worked as the only designer to redesign the major workflow and design new features for a metric performance tracking and dashboarding platform of a top social media company (Platform M for short in the following story). The redesigned workflow and UI based on the latest design system have been launched successfully.

Team & My Role
  • Worked closely with PM and PO and 2 developers on daily basis.
  • Wore multiple hats to define use cases, conduct QA testing, and lead the CSS enhancement in addition to my design work.
Toolbox
  • Figma
  • Design System
  • React / CSS

Challenge

Current metric management and dashboarding experience is inefficient which makes the client team hesitant to introduce the tool to larger user groups.

  • Complex metric adding and dashboard creation process
  • Lack of communication system to notify metric or project risk
  • Generic homepage adding efforts for accessing personal projects

I believe I would be the only one who will create a dashboard like this because I am the admin...I need to make sure the tool is easy to use to be adopted by more users.

With this insight from the PO/client, who is also the key persona, the team believes the users' life could be much easier. Thus, I was asked to design for:

How might we create a simplified metric and project management experience that can enhance users' work efficiency and communicate metric performance risk effectively?

Solution Overview

Simplified workflow to create project and view dashboard

Import multiple metrics at a time and then click one button to view the dashboard.

Proactive metrics tracking via a goal and alert system

Set goals and alerts with the support of real-time calculated instances and then stay informed about risks.

Personalized homepage with restructured navigation

Access my projects, subscriptions, and pinned projects easily and customize homepage as a pinned project.

Process

As the only designer on the team, I was involved in the whole process from gathering requirements to brainstorming, design, tests, and development.

I was able to build empathy with the user and tested the design with some colleagues, despite the time constraints and the nature of the projects, which gave me a limited resource of research and access to end-users. I had conversations with the PO, who is also the most active user, to understand the user behavior/frustration, and also observe how users set up the metrics and goals on the existing platform.

Persona

The initial user group mainly comes from one team that focuses on tracking metrics in a specific area. In the future, there might be more users from different teams to track the performance of other types of metrics.

Main persona: project / metric editors

These groups of users are engineers/data scientists/program managers with a technical background. They use the tool to add metrics, goals, and alerts, and monitor metrics.

  • They may conduct these activities only a few times a year, but every time, they have deep interactions with the tool, which requires an easy-to-follow logic setting /calculation process.
  • They have a clear idea of the metrics and goals they need. A process to handle multiple items together is needed to avoid repetitive clicks.
  • They tend to use a large screen to take a look at "red" metrics and their trends quickly.
  • They are very busy, so they need to finish the editing process efficiently and also have a way to be notified about metric risk without coming to the site.
Secondary persona: dashboard viewers

This group of users are leads or other stakeholders who may be either expert in data or not.

  • They only view dashboard during presentations, when receiving a dashboard shared by others or check metric performance occasionally.
  • The dashboard needs to deliver a quick and clear insight to them.

User Journey

Design

Goal: Create projects and dashboard easily
Insight
  • Multiple clicks have to be done to add metrics one by one while users have a clear idea about all metrics they want to include.
  • Repetitive process of data import and viewership management is required because the dashboard is a separate entity with the project.
Solution
  • Add multiple metrics at one time

    • Use collapsable rows to avoid long-scroll and update metric name while users make edits to better distinguish metrics.

  • Switch between project management and dashboard viewing with one button

    • Cut the separate flow of dashboard creation and management and give users an immediate view about what the dashboard looks like once basic project elements are set up.

Goal: Set goals and alert logics in an intuitive flow
Insight
  • Complex logics exist for different types of goals and ways of setting goal value and also alerts which are based on goals/progress toward goals.
  • Multiple use cases are needed for sending alerts to track metric progress against goals.
  • Technical constraints make it hard to display the goal / alert line while setting.
Solution
  • Present the step and logic based on the mental modal of defining a range on a chart

    • Help users first specify desired performance direction (above or below the goal line) and then define where the goal line is

  • Provide immediate calculated result during goal and alert setting

    • Show the calculated goal during the input process to help users review whether the goal makes sense; display immediately the calculated next instance based on goal value to give users a concrete idea about the ruleset.

  • Use a sentence format to support alert setting

    • Frame the alert rule-setting as "If...when...then" to support add one rule for all goals to avoid repetitive works.

Goal: Get access and organize "my" projects easily
Insight
  • Generic list on the homepage requires searching and saving projects manually from the users.
  • Low usage of some tabs like metrics tab causes a waste of space.
Solution
  • Make homepage a customized personal workspace

    • Structure the information architecture based on "owned by me" and "shared with me". Provide a central portal on the homepage for users to access everything about them quickly. Enable pinned projects and customizable the homepage to help users access the most important projects easily.

  • Help users quickly understand the risk and priorities of projects

    • The color of the KPI and alert related information make the homepage also a dashboard to understand the status across projects.

Test & Iteration

There are multiple rounds of iteration based on feedback from developers and usability testings. The major iterations focus on:

  • Define the information architecture and flow of creating projects and dashboard.
  • Lead the user step by step to set the goals and help them understand the concept of goal setting.
  • Help users understand that alerts are recurring which apply to all goals rather than alerts needed to be set for each goal.

Impact & Take-aways

The redesigned workflow significantly reduced the steps and time users need to set up the projects and dashboards. More user groups will start to use this tool.

With the constraints of limited research resources, I took the hypothesis approach and validated my understanding and design with stakeholders. At the same time, I was trying my best to break these constraints by involving my colleagues to test the design.

I built a collaborative environment by involving the PM and developers, and another designer colleague to brainstorm and critique to avoid design in the silo. As the only designer on the team, I needed to absorb different opinions but at the same time held the designer's point of view to speak for UX.

I enhanced the collaboration between designer and dev by frequently checking in with devs, providing design documents for clear communication, and providing as many details as possible to make CSS enhancement easy.
