Interaction Design | UX Research

Introduction

In summer 2017, I worked as the first and the only UX designer & UX researcher on the SHRINE team in Harvard Catalyst, Harvard Medical School. I was tasked with doing user research and redesigning the Data Steward Application, a web application for Data Stewards to check if there is malicious query when medical researchers access patient data.

Challenge

Medical researcher
Build queries with terms under query topics in SHRINE query tool

Data Steward
Review researchers’ query with Data Steward Applicaion

How can we help the Data Steward identify malicious query behavior?

Here are some types of typical malicious query:

  • Off-topic terms: the terms used to build queries are not consistent with the query topic.
  • Sensitive terms: disease, such as HIV and abortion, usually should not be queried.
  • Re-identification: a user may re-identify a specific patient with a rare condition by using same terms repeatedly in over 15 queries.
  • Bot-attack: if a bot attacks the system, there will be many queries in one minute from the same user.

There was an existing Data Steward Application (DSA). However, the information there was not enough for decision making and there were usability issues due to the lack of UX involvement in development. Also, the users had limited experience and knowledge because the rold of Data Steward (DS) was new.

Solution Overview

Overview recent queries run by researchers

Prioritize the queries to identify abnormal number quickly

Understand the distribution of query terms

Find off-topic terms quickly and get in-context help easily

Review how the query is built in details

Check if there is any pattern indicating re-identifaction, mark and share a problematic query topic

Approach

I adopted the user-centered design process and worked closely with the PM, developers, content curator, and my mentor, the senior UX designer in Harvard Catalyst.

Research

To understand Data Stewards’ behavior and expectation when reviewing the queries, I interviewed 4 Data Stewards (DSs) from pilot participating institutions and observed how they reviewed normal and problematic queries.

Affinity wall to analyze the interview result

Findings
  • Data Stewards (DSs) did not have a deep understanding of their responsibilities and malicious query behavior because they have not started work yet.
  • DSs liked the ability to see the aggreage number of query term frequency, but the terms are structured in folders and DSs feel frustrated to expand the folders with too many clicks.
  • DSs hoped to see how queries are built with terms, in addition to the aggregate data.
  • DSs needed more functions to flag suspicious users or share users’ queries.
Persona

Based on the findings above, I built personas to identify users' pain points and needs.

User Journey

Based on the interview findings and the evaluation on the existing DSA, I created the user journey map to understand the gap between the actual experience and the ideal user flow.

Scope

Based on the journey map, I further defined the project scope, which is to help the Data Stewards (DSs) complete the following tasks more quickly and easily:

  • Prioritize researchers.
  • Find off-topic terms and bot attack by reviewing the query terms with ease.
  • Find patient re-identification by seeing how terms are used in individual queries.
  • Flag/share problematic queries/researchers.

Design

Goal 1: Help Data Stewards prioritize queries
Problem

The previous Data Steward Application (DSA) displays the total number of queries of recent active researchers. However the Data Stewards usually review queries by researchers' query topics. They need more cues to decide which researcher's topic to review first.

Before

Solution
  • Display the number of queries for each researcher's each query topic.

    • Users can first review a query topic which has larger number of queries because this may indicate re-identification or Bot Attack.

After

Goal 2: Help Data Stewards find off-topic terms without the effort of clicks
Problem

The previous Data Steward Application (DSA) displays the query term and its category in a tree view to present their position on the ontology tree. Users need to expand multiple folders to see the query term frequency, which increases the users' workload.

The Previous DSA

Ontology Tree in SHRINE query tool

Solution
  • Display query frequency and category of query terms in a table view.

    • The sequence of the terms is 1st level category - 2nd level category - 3rd level category - query term in alphabetic order so that the structure of categories can be displayed clearly.

After

Goal 3: Help Data Stewards find patient re-identification by seeing the terms in individual queries.
Problem

The Data Stewards hope to see how queries are built in SHRINE query tool to find re-identification, which cannot be done with the previous Data Steward Application.

Query builder in SHRINE query tool

Query report in SHRINE query tool

Solution
  • Present the terms in the way (query viewer) how they build queries in SHRINE query tool, which DSs are familiar with.

    • The boolean between each column is “OR” and the boolean within a cell is “AND”. The modifier means the criteria, such as date range of hospital visit, number of occurrences.

  • Provide a search box to highlight a term.

    • Users can search for and highlight a frequently used term, and see how that term is used with other terms across queries.

    Users can find re-identification by seeing a specific pattern in the usage of that term.

After - Query View

Goal 4: Enable Data Stewards flag/share/contact a suspicious researcher
Problem

The Data Stewards want to mark a suspicious researcher, share the queries of a suspicious researcher to someone who has more knowledge than them to see if the queries are problematic.

Solution
  • Provide mark, contact, and share functions.
Goal 5: Provide cues about how to find potential malicious behavior
Problem

The role of Data Steward was very new. They didn’t know clearly what kinds of malicious behavior they need to look for. They did not have sufficient knowledge about the complex pattern of terms that indicate re-identification.

Solution
  • Provide in-context help about how to find potential malicious behavior.
  • Use red text for sensitive terms and rare diseases as a cue for potential problems.

Test & Iteration

Revision 1: Focus on reviewing queries rather than researchers
Findings from Test
  • Users jumped directly to the aggregate view and query view, which means they care about the task of reviewing queries most.
  • Uses did not notice to query topic, query intent, which are supposed to be helpful for their review.
  • Users did not pay much attention to the background information of a researcher.
  • Users felt confused about whether to flag a researcher or query topic. A flagged query topic is displayed as a flagged researcher in the "review queries" table.
Iteration
  • Use a query topic as the key column of the "reveiw queries" table and the title of the review page. Reduce unecessary information in the review page.

    • This helped users focus on their main task, which is reviewing queries, and prompted them to pay attention to the query topic and intent before reviewing quereis.

    This mades sure the consistency of the flagging interaction. When a user flagges a query topic, it will also be mapped to the status in the "review queries" table.

  • Change the “flag” to "mark" and use a warning icon to avoid confusion with the flag function in SHRINE query tool.

Revision 2: Redesign the sequence of query term frequency
Findings from Test
  • Users prefer to view the query term based on their freqency rather than viewing them based on the ontology hierarchy.
Iteration
  • Sort the query terms in each 1st level category (demographic, diagnosis, medication, etc.) by frequency

    • This help users understand the distribution of terms more quickly while still viewing the context (i.e., category, hierarchy) of the query term.

Round 1

Final Version

Revision 3: Make the search/highlight function more understandable
Findings from Test
  • Users did not pay much attention to the search box to highlight a query term. They also did not have a clear idea about what to search.
Iteration
  • Change the search box to a drop-down list, which provides a list of terms sorted by frequency.

    • The list can prompt the users to check the usage of the frequently used terms and see whether there is a specific pattern that indicates re-identification.

Take-aways

The biggest challenge of this project is to design for users who are not aware of their role and don't have sufficient knowledge about malicious behavior. How to start the design without enough input from the users?

It was very helpful for me to collaborate with the team members who have the expertise about malicious query behavior and medical terminology to gain that knowledge. I took the initiative to have meetings with the technical lead, the content curator, and the PM, making sure my understanding is correct, brainstorming with them, and asking for feedback about my design. This helped me come up with the design that simplifies the query review process by providing the key information and more cues about malicious query behvavior.
